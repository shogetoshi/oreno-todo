# 実装計画: 0053-会議でもログ出力

## 概要

カレンダーイベント（MTG）の開始時にもlog-timer-startプラグインに通知を送り、Todoタスクと同様にLogseq形式でログ出力できるようにする。

## 現状分析

### 既存のプラグイン通知フロー（Todoタスク）

1. **View層** (`TodoItem.tsx`)
   - `handleTimerClick()`が呼ばれる
   - Todoの場合: `onStartTimer(todoId)`を呼び出し

2. **Controller層** (`useTodos.ts`)
   - `startTimer(id)`関数が実行される
   - タイマー開始前にListItemデータを取得: `const targetItem = todos.find(...)`
   - タイマー開始処理: `setTodosWithPersist((prev) => TodoRepository.startItemTimerExclusive(prev, id))`
   - **プラグインに通知**: `window.electronAPI.notifyTimerStart(targetItem.toJSON())`

3. **Electron Preload** (`preload.ts`)
   - `notifyTimerStart: (itemData) => ipcRenderer.invoke('notify-timer-start', itemData)`

4. **Electron Main Process** (`main.ts`)
   - `ipcMain.handle('notify-timer-start', async (_, itemData) => { ... })`
   - `pluginManager.notifyTimerStart(itemData)`を呼び出し

5. **PluginManager** (`pluginManager.ts`)
   - `notifyTimerStart(itemData)`メソッドが実行される
   - 全てのプラグインの`onTimerStart`ハンドラーを呼び出し
   - コンテキストオブジェクトを作成して渡す

6. **プラグイン** (`log-timer-start.js`)
   - `onTimerStart(context)`が実行される
   - `context.data`にListItemのJSONデータが格納されている
   - Logseq形式でファイルに記録

### 現在のカレンダーイベント開始フロー

1. **View層** (`TodoItem.tsx`)
   - `handleTimerClick()`が呼ばれる
   - カレンダーイベントの場合: `onStartCalendarEvent(todoId)`を呼び出し

2. **Controller層** (`useTodos.ts`)
   - `startCalendarEvent(id)`関数が実行される
   - MTG URLを開く: `await openMeetingUrl(id)`
   - 他の実行中タスクを停止 + カレンダーイベントを完了状態にする
   - **プラグインへの通知が行われていない** ← 今回追加する部分

## 課題

カレンダーイベントの開始時（`startCalendarEvent`関数）で、プラグインへの通知が行われていないため、log-timer-startプラグインがカレンダーイベントをログに記録できない。

## 実装方針

### MVCアーキテクチャの遵守

#### Model層
- 変更なし（既存のCalendarEventクラスとTodoRepositoryはそのまま使用）

#### Controller層
- `useTodos.ts`の`startCalendarEvent`関数を修正
- Todoタスクの`startTimer`関数と同様に、プラグインに通知を追加

#### View層
- 変更なし（既存のUIフローをそのまま使用）

#### Electron層（Main Process/Preload）
- 変更なし（既存のIPC通信とプラグインマネージャーをそのまま使用）

### プラグイン層
- 変更なし（log-timer-start.jsは既にListItemインターフェースで実装されており、typeフィールドで判別可能）

## 実装内容

### 1. Controller層の修正

**ファイル**: `src/hooks/useTodos.ts`

**変更箇所**: `startCalendarEvent`関数

**変更内容**:
```typescript
const startCalendarEvent = useCallback(async (id: string) => {
  // プラグインに通知するために、開始前のListItemデータを取得
  const targetItem = todos.find(item => item.getId() === id);

  // 1. MTG URLを開く
  await openMeetingUrl(id);

  // 2. 他の実行中タスクを全て停止
  // 3. カレンダーイベントを完了状態にする
  setTodosWithPersist((prev) => {
    const itemsWithStoppedTimers = TodoRepository.stopAllRunningItems(prev);
    return TodoRepository.completeItem(itemsWithStoppedTimers, id);
  });

  // 4. プラグインにカレンダーイベント開始を通知（非同期、エラーは無視）
  if (targetItem) {
    window.electronAPI.notifyTimerStart(targetItem.toJSON()).catch(error => {
      console.error('Failed to notify plugins:', error);
    });
  }
}, [openMeetingUrl, setTodosWithPersist, todos]);
```

**変更のポイント**:
- `startTimer`関数と同じパターンでプラグインに通知
- カレンダーイベントの開始前にListItemデータを取得
- プラグイン通知は非同期で実行し、エラーが発生してもメイン処理に影響しない
- `todos`を依存配列に追加（targetItemの取得に必要）

### 2. プラグイン側の対応

**ファイル**: `sample-plugins/log-timer-start.js`

**変更**: 不要

**理由**:
- プラグインは既に`context.data.type`フィールドでTodoとCalendarEventを判別可能
- `context.data`にはListItemのJSON表現が格納される
- CalendarEventの場合、以下のフィールドが追加で含まれる:
  - `type: 'calendar_event'`
  - `startTime`: イベント開始時刻
  - `endTime`: イベント終了時刻
  - `meetingUrl`: MTG URL
- 現在のlog-timer-startプラグインはtypeフィールドを使用していないが、将来的に必要に応じて分岐処理を追加可能

## リスク評価

### 低リスク
- **変更範囲が限定的**: Controller層の1つの関数のみを修正
- **既存のパターンを踏襲**: `startTimer`と同じ実装パターンを使用
- **後方互換性**: 既存のプラグインはtypeフィールドを無視するため影響なし

### 注意点
- `todos`を依存配列に追加するため、`startCalendarEvent`関数の再生成頻度が増える
  - ただし、`useCallback`を使用しているため、実際のパフォーマンス影響は軽微
  - `startTimer`関数も同様の依存配列構成であり、問題ないことが実証済み
