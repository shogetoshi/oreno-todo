# Issue #0054: カレンダーイベントの上書き抑止 - 実装計画書

## 概要

カレンダーイベントを複数回取得した際、特定の条件下では上書きを抑止するロジックを追加する。これにより、ユーザーが手動で記録した実行時間や完了状態が意図せず失われることを防ぐ。

## Issue内容の再確認

**上書きしない条件**:
- すでに`timeRanges`に値が入っている場合
- 完了になっている場合

**上書きする条件**:
- それ以外（`timeRanges`が空かつ未完了）

### 2. 現在の上書き挙動

**場所**: `src/models/TodoRepository.ts` の `addCalendarEventsToItems()` メソッド（302-317行目）

```typescript
static addCalendarEventsToItems(
  items: ListItem[],
  events: GoogleCalendarEvent[],
  projectRepo?: ProjectDefinitionRepository
): ListItem[] {
  const newEvents = this.createCalendarEventsFromGoogleEvents(events, projectRepo);

  // 新しいイベントのIDセットを作成
  const newEventIds = new Set(newEvents.map(event => event.getId()));

  // 既存のアイテムから、新しいイベントと同じIDを持つものを除外
  const existingItemsWithoutDuplicates = items.filter(item => !newEventIds.has(item.getId()));

  // 既存のアイテム（重複除外済み） + 新しいイベント
  return [...existingItemsWithoutDuplicates, ...newEvents];
}
```

**問題点**: 同一IDのイベントが存在する場合、無条件で新しいイベントで上書きしている。

### 4. IDの生成ロジック

**場所**: `CalendarEvent.generateIdFromEvent()` メソッド（389-401行目）

```typescript
private static generateIdFromEvent(event: CalendarEventType): string {
  const startTime = event.start.dateTime || event.start.date || '';
  const createdTime = event.created;
  const uniqueString = `${startTime}_${createdTime}`;

  const hash = this.simpleHash(uniqueString, false);
  const hash2 = this.simpleHash(uniqueString, true);

  const hashHex1 = Math.abs(hash).toString(16).padStart(8, '0');
  const hashHex2 = Math.abs(hash2).toString(16).padStart(8, '0');

  return `cal-${hashHex1}-${hashHex2.substring(0, 4)}-${hashHex2.substring(4, 8)}-${hashHex1.substring(0, 12)}`;
}
```

**重要**: 同じGoogleカレンダーイベント（開始時刻+作成時刻が同一）からは常に同じIDが生成される。

### 実装の詳細設計

#### 1. 上書き判定ロジックの追加

**ファイル**: `src/models/TodoRepository.ts`

**メソッド**: `addCalendarEventsToItems()`

**変更内容**:

```typescript
static addCalendarEventsToItems(
  items: ListItem[],
  events: GoogleCalendarEvent[],
  projectRepo?: ProjectDefinitionRepository
): ListItem[] {
  const newEvents = this.createCalendarEventsFromGoogleEvents(events, projectRepo);

  // 既存のCalendarEventをIDでマッピング
  const existingEventsMap = new Map<string, ListItem>();
  items.forEach(item => {
    existingEventsMap.set(item.getId(), item);
  });

  // 新しいイベントと既存イベントをマージ
  const mergedEvents: ListItem[] = newEvents.map(newEvent => {
    const existingEvent = existingEventsMap.get(newEvent.getId());

    // 既存イベントが存在しない場合は新規追加
    if (!existingEvent) {
      return newEvent;
    }

    // 既存イベントがCalendarEventでない場合は上書き（安全のため）
    if (existingEvent.getType() !== ListItemType.CALENDAR_EVENT) {
      return newEvent;
    }

    const existing = existingEvent as CalendarEvent;

    // 上書き判定: timeRangesに値がある、または完了している場合は既存を保持
    if (existing.timeRanges.length > 0 || existing.isCompleted()) {
      return existing;
    }

    // それ以外は上書き
    return newEvent;
  });

  // 新しいイベントで上書きされなかった既存アイテムを抽出
  const newEventIds = new Set(newEvents.map(event => event.getId()));
  const remainingItems = items.filter(item => !newEventIds.has(item.getId()));

  // 残った既存アイテム + マージされたイベント
  return [...remainingItems, ...mergedEvents];
}
```

#### 2. 補助メソッドの追加（オプション）

可読性向上のため、上書き判定ロジックを独立したメソッドとして切り出す。

**メソッド名**: `shouldOverwriteCalendarEvent()`

```typescript
/**
 * カレンダーイベントを上書きすべきかどうかを判定する
 * @param existingEvent 既存のCalendarEvent
 * @returns 上書きする場合はtrue、既存を保持する場合はfalse
 */
private static shouldOverwriteCalendarEvent(existingEvent: CalendarEvent): boolean {
  // timeRangesに値がある場合は上書きしない
  if (existingEvent.timeRanges.length > 0) {
    return false;
  }

  // 完了している場合は上書きしない
  if (existingEvent.isCompleted()) {
    return false;
  }

  // それ以外は上書き
  return true;
}
```

## エッジケース・考慮事項

### 1. データ型の整合性

- **問題**: 既存アイテムがCalendarEventでない場合
- **対応**: `getType()`で型チェックを行い、CalendarEventでない場合は上書きする

### 2. timeRangesの値検証

- **問題**: `timeRanges`が`undefined`や`null`の可能性
- **対応**: `timeRanges.length > 0`のチェックで暗黙的に安全に扱える（配列のlengthプロパティ）

### 3. IDの衝突

- **問題**: TodoとCalendarEventで偶然IDが衝突する可能性
- **対応**: CalendarEventのIDは`cal-`プレフィックス付き、TodoはUUID形式なので衝突は発生しない

### 4. パフォーマンス

- **問題**: イベント数が多い場合のパフォーマンス
- **現状**: Googleカレンダーからの取得上限は100件、Mapを使った探索はO(1)なので問題なし

### 5. 部分的な更新

- **問題**: イベントのタイトルや時間が変更された場合、更新したいが完了状態は保持したい
- **対応**: 現在の仕様では、完了またはtimeRangesがある場合は完全に保持する。部分更新が必要な場合は、将来的に`mergeCalendarEvent()`のようなメソッドを追加することを検討
